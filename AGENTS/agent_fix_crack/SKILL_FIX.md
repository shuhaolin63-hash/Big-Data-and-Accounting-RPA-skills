# RPA 破障修复智能体（Crack / Fix Agent）

## 核心定位

双场景ID合规处理。有源码清旧ID，无源码等用户自填ID，杜绝串号、泄露、风控。

---

## 触发场景

| 场景 | 说明 | 触发条件 |
|------|------|----------|
| 复用他人工程 | 全自动扫描清除他人Base64硬编码ID | 用户导入外部源码工程 |
| 从零生成新作业 | 预留ID注入空位，等待用户自填 | 系统生成新作业工程 |
| ID合法性校验 | 检测ID真实性、版本号、学生账号有效性 | 用户输入ID后需要验证 |
| 规避风控 | 检测是否存在篡改风险、串号风险 | 提交作业前的最终校验 |

---

## 双模式工作流

### 模式A：有外部源码 -> 全自动清除旧ID

```
Step1 接收用户导入的源码工程
Step2 全自动扫描：
  - 遍历所有.py/.json文件
  - 正则匹配Base64编码和明文ID模式
  - 提取所有需要清除的字段
Step3 执行清除：
  - 解码Base64 -> 清空敏感字段 -> 重新编码写回
  - 替换.py中的硬编码值为PLACEHOLDER
  - 清空.variable.json中的默认ID值
Step4 输出清洗报告：
  - 清除了哪些文件中的哪些字段
  - 剩余待用户补充的ID空位
Step5 交由老师Agent继续
```

### 模式B：无外部源码 -> 预留ID注入空位

```
Step1 接收资产Agent生成的工程框架
Step2 在所有需要ID的位置预留注入点
Step3 生成ID填写指引 -> 输出id_inject_guide.txt
Step4 等待用户通过runtime主动注入
Step5 注入前进行合法性校验
Step6 校验通过 -> 动态注入 -> 生成最终作业
Step7 校验不通过 -> 返回错误原因
```

## ID清除流程

### 扫描范围

| 文件类型 | 扫描内容 | 处理方法 |
|----------|----------|----------|
| `.py` | `user_id`、`task_id`、`auth`赋值语句 | 替换为 `PLACEHOLDER` |
| `.json` | token、auth的Base64编码字段 | 解码 -> 清空 -> 重新编码 |
| `rzgc.variable.json` | 全部Base64编码存储 | 解码 -> 清空所有ID字段 -> 写回 |
| `variable.json` | `defaultValue`中的ID | 清空默认值 |
| `ctrl.json` | URL中的硬编码username | 保留URL结构，清空用户参数 |

### ID检测模式

```
检测正则表达式：
1. Base64编码模式: [A-Za-z0-9+/]{20,}={0,2}
2. UUID模式: [0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}
3. 32位hex模式: [0-9a-f]{32}
4. 字段名模式: user_id|task_id|UserID|TaskID|userId|taskId
```

### 清洗报告格式

```
[ID清洗报告]
扫描文件数: XX
发现ID字段: XX
已清除: XX
未清除（需手动）: XX

已清除详情:
  文件1: 清除了字段A, 字段B
  文件2: 清除了字段C

待处理:
  文件3: 字段D需要用户补充
```

## 风控校验流程

### 校验项

| 校验项 | 检查内容 | 不通过处理 |
|--------|----------|-----------|
| ID格式 | 是否符合UUID/32位hex格式 | 提示格式错误 |
| ID长度 | UserID 36位(含连符号)、TaskID 32位 | 提示长度异常 |
| 编码检测 | 是否存在Base64编码的其他ID | 自动清除并警告 |
| 重复检测 | 是否与系统已有工程ID重复 | 提示串号风险 |
| 篡改检测 | 代码中是否有非标准修改痕迹 | 标记风险项 |

### 校验流程

```
用户提交ID
    │
    ▼
┌──────────────┐
│  格式检查     │  <-- UUID/32位hex格式验证
└──────┬───────┘
    │ 格式错误
    ▼
┌──────────────┐
│  长度检查     │  <-- 36位/32位长度验证
└──────┬───────┘
    │ 长度异常
    ▼
┌──────────────┐
│  编码检测     │  <-- 扫描残留Base64编码
└──────┬───────┘
    │ 发现残留
    ▼
┌──────────────┐
│  重复检测     │  <-- 检查是否与已有ID重复
└──────┬───────┘
    │ 发现重复
    ▼
┌──────────────┐
│  篡改检测     │  <-- 检查代码非标准修改
└──────┬───────┘
    │ 发现篡改
    ▼
┌──────────────┐
│  生成风控报告  │
└──────┬───────┘
    │
    ▼
输出校验结果（通过/不通过+风险项）
```

## 与Teacher Agent的协作关系

```
┌─────────────┐    清理后代码     ┌─────────────┐
│  破障Agent  │ ──────────────→  │  老师Agent  │
│  (清旧ID)   │                  │  (讲框选)   │
└──────┬──────┘                  └─────────────┘
       │ 注入用户ID
       ▼
┌─────────────┐
│  CORE_RUNTIME │
│  (运行注入)  │
└─────────────┘

协作流程：
1. 破障Agent先扫描清除所有硬编码ID（有源码场景）
2. 或等待用户通过runtime_id_input.py注入ID（无源码场景）
3. 清理完成后将代码交给老师Agent进行框选教学
4. 最终由CORE_RUNTIME运行注入用户ID

关键约束：
- 破障Agent必须在老师Agent之前执行（有源码场景）
- 破障Agent必须在老师Agent之后执行（无源码场景，最后注入ID）
```
